### 2.2 VP DB 구조 발견 및 대응

**기존 가정 (Plan 문서):**
- `sol_VP_rebalancing_inform`에 ISIN, weight 컬럼이 있을 것

**실제 DB 구조:**
- `sol_VP_rebalancing_inform`: 리밸런싱 이벤트 로그만 (ISIN/weight 없음)
  - 컬럼: 리밸런싱날짜, 펀드설명, port('MP'/'VP'), rebalancing_reason, version, 경기국면
- VP 보유종목: `sol_DWPM10530` (VP 전용 펀드코드로 조회)
- VP 기준가: `sol_DWPM10510` (VP 전용 펀드코드로 조회)

**VP 전용 펀드코드 매핑 (`_FUND_DESC_TO_VP_CODE`):**

| fund_desc | VP 코드 | VP 펀드명 |
|-----------|---------|----------|
| MS GROWTH | 3MP01 | MySuper 성장형 MP |
| MS STABLE | 3MP02 | MySuper 안정형 MP |
| TDF2050 | 1MP50 | TDF 2050 MP |

### 2.3 VP 로딩 함수 구현 (`data_loader.py`)

| 함수 | 용도 | 데이터소스 |
|------|------|-----------|
| `load_vp_rebal_date(fund_desc)` | VP 최근 리밸런싱 날짜 | sol_VP_rebalancing_inform |
| `load_vp_holdings_8class(vp_code)` | VP 보유종목 → 8분류 집계 | sol_DWPM10530 + universe_non_derivative |
| `load_vp_weights_8class(fund_desc)` | fund_desc → VP 코드 → 8class | 위 함수 래핑 |
| `load_vp_nav(fund_desc_or_code)` | VP 기준가 시계열 | sol_DWPM10510 |

VP 비중 집계 방식:
1. `_FUND_DESC_TO_VP_CODE`로 VP 코드 결정
2. `sol_DWPM10530.NAST_TAMT_AGNST_WGH` (순자산비중) 사용
3. `ITEM_CD`(ISIN) → `universe_non_derivative` 방법3 → 8분류 매핑

### 2.4 Tab 2 DB 연동

**교체 항목:**

| 항목 | Before (mockup) | After (DB) |
|------|-----------------|------------|
| VP 비중 | `[26.0, 29.0, ...]` | `load_vp_weights_8class()` or FUND_MP_DIRECT |
| AP 비중 | `[25.3, 30.1, ...]` | `_tab1_hold.groupby('자산군')['비중(%)'].sum()` |
| 리밸런싱일 | `"2026-02-03"` | `load_vp_rebal_date()` → sol_VP_rebalancing_inform |
| 추적오차 | `"0.15%"` | AP NAV vs VP NAV std × √252 × 100 |
| 누적수익률 VP | `make_nav(1000, ...)` | `load_vp_nav()` → sol_DWPM10510 |
| MP 비중 | 이미 Phase 2에서 연동 | 변경 없음 |
| Gap 추이 | random walk | mockup 유지 (일별 역사적 VP 비중 구축 필요) |

**VP 로딩 우선순위:**
1. FUND_MP_DIRECT → VP = MP (사모펀드, DB에 VP 없음)
2. FUND_MP_MAPPING → `load_vp_weights_8class(fund_desc)` → DB
3. fallback hardcoded

**DB 소스 표시:** VP DB 로드 성공 시 "📡 DB 데이터" / 실패 시 "⚠️ 목업 데이터"
