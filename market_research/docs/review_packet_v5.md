### 2.1 사례 1: 시제 변조 ("유력" → "확정")

**발생 조건**:
- debate 실행: 2026-04-10T14:38
- evidence에 포함된 기사: "이창용 마지막 금통위, 7연속 동결 **유력**" (04-09, 금통위 **전날**)

- 04-10 동결 확정 기사 158건이 DB에 있었으나 중요도 순위에서 밀려 evidence TOP 15에 미포함

**Opus 출력**:
```
"국내에서는 한국은행이 7회 연속 기준금리를 동결한 가운데" [ref:5]
```

**문제**:
- evidence에는 **"유력"** 기사만 있는데 **"동결한"**으로 확정형 서술
- Haiku 에이전트(quant)가 "한은 7연속 동결 유력"이라고 응답 → Opus가 확정형으로 변환
- LLM이 출처의 시제를 무시하고 자체 판단으로 사실을 확정

**근본 원인**: Opus가 에이전트 응답을 종합할 때 출처 기사의 원문 표현을 확인하지 않고, 에이전트 요약만 보고 서술. "유력"이라는 불확실성 표현이 종합 과정에서 소실됨.

## 4. 후처리 체계

### 4.1 `_sanitize_customer_comment(text, indicators, annotations)`

| 단계 | 검증 | 처리 | 상태 |
|------|------|------|------|
| 1. 내부 토큰 제거 | `[ref:N]`, 살리언스, `F_[A-Z_]+` 등 | **자동 제거** | 구현 완료 |
| 2. raw number 경고 | `[+-]\d{3,}\.\d+` (부호+대형숫자) | 경고 | 구현 완료 |
| 3. 펀드 액션 패턴 | "당사는", "비중을 확대" 등 주어+동사 | 경고 | 구현 완료 |
| 4. 권고형/처방형 | "줄일 필요가 있다", "적절하다" 등 | 경고 | **신규 구현** |
| 5. 역수익률곡선 검증 | 2Y < 10Y이면 "역수익률" 금지 | 경고 | 구현 완료 |
| 6. **시제/불확실성 검증** | evidence "유력" → 코멘트 "확정형" 탐지 | 경고 | **신규 구현** |
| 7. **ref ↔ 문장 키워드 교차검증** | 문장 토픽 ↔ ref 기사 토픽 불일치 | 경고 | **신규 구현** |

### 4.2 자동 제거 vs 경고 정책

| 분류 | 대상 | 처리 |
|------|------|------|
| **A. 자동 제거** | [ref:N], 내부 metric, 내부 코드명, inline citation | 코드가 제거 |
| **B. 경고** | raw number, 펀드 액션, 권고형, 사실 오류, 시제 불일치, ref 오매핑 | admin 검수 |
| **C. admin 필수 확인** | 시제 validator 경고, ref mismatch, 당일 확정 이벤트 관련 | admin 화면 표시 |

### 4.3 금지 패턴: 단어가 아닌 "주어+동사" 패턴

```
펀드 액션 금지: "당사는 비중을 확대한다" → 탐지
시장 설명 허용: "듀레이션 부담이 확대되고 있다" → 통과
권고형 탐지: "듀레이션을 줄일 필요가 있다" → 경고
```

### 4.4 시제/불확실성 validator (신규)

```
evidence 불확실 표현: 유력, 전망, 무게, 가능성, 관측, 예상, 시사, 검토, 논의 등
코멘트 확정 표현: 동결한, 결정했, 확정됐, 발표했, 합의했 등
→ evidence가 불확실 + 코멘트가 확정 = 경고
```

테스트 결과: "동결 유력" evidence + "동결한 가운데" 코멘트 → **경고 발생 확인**

### 4.5 ref ↔ 문장 키워드 교차검증 (신규)

```
문장 키워드: 한국은행, 금통위, 기준금리
ref 기사 키워드: IMF, growth, prices
→ 토픽 불일치 = 경고
```

테스트 결과: 한국은행 문장 + ref:5(IMF 기사) → **ref 오매핑 경고 발생 확인**

검증 테스트:
- 시장 설명 문장 → warnings 0건 ✓
- 펀드 액션 문장 → 7건 탐지 ✓
- 기존 4월 코멘트 → 5건 탐지 ✓

### 4.3 미구현 — hallucination 대응 검증

| 검증 | 설명 | 상태 |
|------|------|------|
| ref:N ↔ 문장 키워드 교차 | 문장 핵심어와 ref 기사 제목 비교, 불일치 시 경고 | **미구현** |
| 시제 검증 | "확정"형 서술 + ref 기사가 "유력/전망" → 경고 | **미구현** |

## 7. 리뷰 요청 포인트

### 7.1 Hallucination 방어 (구현 완료, 한계 리뷰 요청)

1. **시제 validator**: rule-based 구현 완료 (불확실 lexicon 18개 × 확정 패턴 15개 교차). 이 방식의 coverage가 충분한가? 놓칠 수 있는 패턴은?

2. **ref 오매핑 validator**: 토픽 키워드 사전(8개 토픽군) 기반 교차검증 구현 완료. client에서는 inline ref 자체를 제거하여 오매핑 노출 원천 차단. 이 "제거" 판단이 맞는가, 아니면 정밀도를 높여 살려야 하는가?

3. **최신 뉴스 슬롯**: 당일 TIER1/TIER2 기사 최대 2건 전용 슬롯 구현. 이 접근이 "중요 확정 이벤트 0건 탈락"을 충분히 방지하는가?

### 7.2 Client/Admin 분리

4. **client inline ref 제거 판단**: ref 오매핑 위험 때문에 client에 번호 노출을 포기하고 하단 출처 목록만 제공하는 것이 적절한가? 기관 고객이 inline citation을 기대하는가?

5. **admin 검수 부담**: 시제 경고 + ref 경고 + 권고형 경고가 동시에 뜨면 admin 피로도가 높을 수 있음. 경고 우선순위나 요약 제시가 필요한가?

### 7.3 산출물 품질

6. **매크로 브리핑 톤**: 펀드 액션/권고형 문장 금지 + 시장 설명 허용의 구분이 실무에서 적절한가?

7. **전체 성숙도**: 실증된 hallucination 3건 + rule-based 방어체계 구현. "client-facing 신뢰성"이 내부 파일럿 수준에 도달했는가?

### 7.4 남은 한계 (P1/P2)

8. **sentence-level evidence trace**: 완전한 문장↔기사 1:1 매핑은 미구현. NLI 도입 시점과 ROI는?
9. **자동 치환**: 시제 불일치 문장을 자동으로 완화 표현으로 치환하는 것이 바람직한가? ("동결한" → "동결 전망이 우세했던")
