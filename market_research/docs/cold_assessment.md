## [3] 냉정한 판단 근거

### 근거 1: 테스트가 0건이다

`market_research/` 전체에 unit test, integration test가 하나도 없다. `tests/ablation_test.py`는 정답 기반 테스트가 아니라 분포 통계 비교 스크립트다. salience 공식의 가중치(0.30/0.25/0.25/0.20)가 바뀌면 downstream 전체가 영향받지만, 이를 잡아줄 regression test가 없다. **실무에서는 코드 한 줄 수정이 전체 출력을 망가뜨릴 수 있고, 그걸 아무도 모른다.**

### 근거 2: LLM 출력의 깨진 토픽명이 방치되어 있다

분류기(Haiku)가 `유가_에너riley`, `AI_반도gheorghe`, `비트코silon_크립토` 같은 깨진 토픽명을 반환한 사례가 실제 데이터에 존재한다. 이 기사들은 정상 토픽과 별개로 집계되므로 토픽 카운트, event clustering, GraphRAG 노드 모두에서 오염된다. **21개 허용 토픽 목록과 대조하는 validation이 `classify_batch`에 없다.** post-processing에서 가장 가까운 정상 토픽으로 매핑하는 코드도 없다.

### 근거 3: 수치 가드레일이 프롬프트 텍스트 지시로만 되어 있다

"제공된 수치를 절대 수정/반올림하지 마세요"라는 규칙이 Quant 에이전트와 Opus 종합 프롬프트에 텍스트로만 존재한다. LLM 출력을 파싱하여 원본 수치와 대조하는 post-processing 검증 코드는 없다. **LLM이 "S&P500 -4.5%"를 "S&P500 -4%"로 반올림하거나, 존재하지 않는 수치를 생성해도 감지할 수 없다.** OCIO 운용보고서에서 수치 오류는 단순 신뢰 하락이 아니라 규제 리스크다.

### 근거 4: 정제 레이어의 가중치/임계치가 실험이 아닌 직관 기반이다

salience 공식의 가중치(0.30/0.25/0.25/0.20), Jaccard 임계치(0.15/0.20), SequenceMatcher 임계치(0.3), hybrid_score의 salience 가중치(0.3), GraphRAG 엣지 가중치 공식(0.3+sal×0.4), anomaly 캡(7일) — 이 모든 파라미터가 코드 주석이나 devlog에 유도 과정 없이 하드코딩되어 있다. **A/B 테스트나 grid search 같은 실험 기반 튜닝의 흔적이 없다.** ablation test가 존재하지만 이것은 파라미터 최적화가 아니라 on/off 비교다.

### 근거 5: monthly rebuild와 daily incremental의 그래프 상태가 분기한다

`build_insight_graph()`(monthly)에서는 Self-Regulating TKG(decay/merge/prune)가 호출되지 않는다. `add_incremental_edges()`(daily)에서는 호출된다. 따라서 한 달간 daily incremental이 누적되면 decay/prune으로 엣지가 정리되지만, 다음 달 monthly rebuild 시 이전 달 그래프를 그대로 로드하므로 decay/prune 결과가 사라진다. **또한 daily incremental에서는 전이경로 precompute가 호출되지 않으므로, 일일 실행 후 전이경로가 stale 상태로 남는다.**

### 근거 6: evidence trace가 입력 수준에서 멈춰 있다

debate 결과에 `_evidence_ids`(15건)가 포함되지만, 이 ID 리스트가 Opus 종합 프롬프트에는 전달되지 않는다. Opus는 4인 에이전트의 텍스트 요약만 받는다. **따라서 최종 코멘트의 특정 문장이 어떤 기사에 기반했는지 역추적이 불가능하다.** "이 문장의 근거가 뭐냐?"라는 질문에 "이 15개 기사 중 하나일 것이다"까지만 답할 수 있다.

### 근거 7: 뉴스 수집의 소스 다양성과 안정성이 취약하다

실제 소스 분포: 네이버검색 55,666건(89%), 나머지 11% — 사실상 단일 소스 의존이다. Finnhub/NewsAPI는 건수가 적고, rate limit 처리(retry/backoff)가 일반 수집 함수에 없다. **네이버 API 변경이나 차단 시 수집량이 89% 감소한다.** NewsAPI 무료 플랜의 약관 위반 가능성이 문서에 명시되어 있지만 대체 소스는 아직 구현되지 않았다.

### 근거 8: 임베딩 모델이 영어 전용인데 기사의 89%가 한국어다

`all-MiniLM-L6-v2`는 영어 전용 모델이다. 코드 주석에도 `영어 뉴스용`이라고 명시되어 있다. 그런데 수집되는 기사의 89%가 네이버 한국어 뉴스다. **한국어 기사의 vectorDB 검색 품질이 보장되지 않으며, `search_for_factors()`에서 한국어 기사가 영문 쿼리로 검색되는 구조적 불일치가 있다.** 한국어 검색 품질 테스트도 없다.

### 근거 9: 모니터링과 알림이 전혀 없다

daily_update가 실패하면 stdout에 에러를 출력할 뿐이다. Slack, 이메일, webhook 알림 연동이 없다. 비용 추적은 comment_engine에만 부분 구현되어 있고, news_classifier와 graph_rag의 LLM 비용은 추적되지 않는다. **하루 수집이 실패해도 아무도 모르고, 한 달 후 report 생성 시 빈 데이터로 코멘트가 나갈 수 있다.**

### 근거 10: 2025년 12개월 데이터가 미분류 상태다

2025-01~12 총 18,980건의 기사가 수집되어 있지만 분류/정제가 0%다. GraphRAG의 이전 달 누적 로직이 2025년 데이터를 참조할 수 있고, 이때 분류/정제 메타가 없는 기사가 투입될 수 있다. **다만 현재 GraphRAG 빌드가 2026-01부터 시작되므로 즉각적 문제는 아니지만, 전체 시계열 분석 시 공백이 된다.**

## [4] 어디까지는 쓸 수 있고, 어디부터는 위험한가

### Case 1: 사람이 마지막에 검토하는 내부 보조도구

**가능한 용도:**
- 월간 시장 리뷰의 초안 생성 (실무자가 수치와 논리를 수정)
- 주요 뉴스 토픽 트렌드 대시보드 (정량 집계)
- debate 결과를 참고 자료로 활용 (합의/쟁점 구조 활용)
- 매크로 지표 수집 자동화 (SCIP/FRED/ECOS)

**위험한 용도:**
- 수치가 포함된 문장을 검토 없이 그대로 사용
- 특정 자산군에 대한 판단을 debate 결과에 의존

**실제 사고 포인트:**
- LLM이 S&P500 수익률 -4.5%를 -45%로 오타내면 검토자가 안 잡을 수 있다 (특히 바쁜 월말)
- GraphRAG 전이경로가 "유가→인플레→ECB"인데, 실제로는 ECB가 유가와 무관한 결정을 했을 수 있다 — LLM 추론의 hallucination
- 3월처럼 지정학 이슈가 지배적인 달에 다른 자산군 분석이 피상적일 수 있다

### Case 2: 거의 무인으로 고객용 산출물을 내는 자동 시스템

**가능한 용도:** 없다. 현재 상태로는 불가.

**위험한 용도:**
- 고객에게 직접 전달되는 운용보고서 코멘트 자동 생성
- 투자 의사결정에 직접 영향을 주는 매크로 전망

**실제 사고 포인트:**
- 수치 가드레일 없음 → LLM이 수치를 변조해도 감지 불가
- 깨진 토픽명 → 토픽별 통계가 왜곡되어 잘못된 시장 진단
- 네이버 API 장애 시 수집 89% 감소 → 빈약한 뉴스로 debate 실행 → 피상적 코멘트
- evidence trace 부재 → 고객이 "이 분석의 근거가 뭐냐"고 물으면 답할 수 없음

## [5] 가장 강한 부분 3개

### 1. 정제 레이어의 계층 분리와 downstream 반영

dedupe → event clustering → salience → fallback이 각각 독립 함수로 분리되어 있고, 결과가 GraphRAG(stratified sampling, salience 가중 엣지), vectorDB(hybrid_score), debate(primary 필터, diversity guardrail)에 실제로 소비된다. **단순히 "정제 코드를 만들었다"가 아니라 downstream 전체가 정제 메타에 의존하도록 배선이 완료되어 있다.** 실무적으로 이것은 "정제 품질을 올리면 모든 downstream이 동시에 개선된다"는 의미이므로, 파이프라인 설계로서 가치가 있다.

### 2. GraphRAG stratified sampling의 설계 판단

200 fixed → topic-stratified dynamic cap(300~500)으로 변경하면서 토픽 다양성 233% 개선, top1 쏠림 28pp 감소를 달성했다. **이 결정은 단순 파라미터 조정이 아니라, "GraphRAG는 요약이 아니라 연결이므로 다양성이 중요하다"는 설계 판단이 반영된 것이다.** 비용도 월 $4→$8 수준으로 통제된다. 검증도 100/200/400 비교를 실제로 수행했다.

### 3. debate의 4인 에이전트 + Opus 종합 구조

Bull/Bear/Quant/monygeek의 역할 분리로 단일 LLM의 편향을 구조적으로 완화한다. 특히 Quant를 Priority Anchor로 지정하여 수치 충돌 시 데이터 우선 원칙을 세운 것, monygeek에 유로달러 학파 프레임워크를 부여하여 주류 해석에 대한 대안적 시각을 강제한 것은 **운용보고서의 "다양한 시각에서 균형 잡힌 분석"이라는 실무 요구를 코드 구조로 반영한 것**이다.

## [6] 가장 취약한 부분 3개

### 1. LLM 출력에 대한 validation이 없다 — 깨진 토픽, 수치 변조, hallucination 방치

분류기가 `유가_에너riley`를 반환해도 그대로 저장된다. Opus가 -4.5%를 -4%로 반올림해도 감지되지 않는다. debate에서 "VIX가 40을 돌파했다"고 말해도 실제 VIX 데이터와 대조하지 않는다. **이 세 가지가 모두 "프롬프트 지시에만 의존하고, post-processing 검증이 없다"는 동일 패턴이다.** OCIO 운용보고서에서 수치 하나가 틀리면 전문성에 대한 신뢰가 무너지고, 규제 보고에서는 법적 리스크가 된다. 이것은 "나중에 고치면 되는" 문제가 아니라, 현재 시스템을 내부 보조도구로 쓸 때도 검토자가 반드시 수치를 전수 확인해야 한다는 의미다.

### 2. 네이버 89% 단일 소스 의존 + 영어 전용 임베딩 모델

수집량의 89%가 네이버검색인데, vectorDB 검색은 영어 전용 모델이다. 이것은 "수집은 한국어, 검색은 영어"라는 구조적 불일치다. **PA 기여도 원인 분석(`search_for_factors`)에서 한국어 기사가 영문 쿼리로 검색되므로, 국내 채권/국내 주식 관련 뉴스 매칭 품질이 보장되지 않는다.** 또한 네이버 API 하나가 막히면 수집 89%가 증발하는 단일 장애점(SPOF)이다.

### 3. 테스트 0건 + 파라미터 근거 부재 = 회귀 방지 불가

salience 가중치, Jaccard 임계치, hybrid_score 가중치 등 10개 이상의 핵심 파라미터가 근거 없이 하드코딩되어 있다. 이 파라미터를 수정했을 때 영향을 검증할 테스트가 하나도 없다. **코드 수정 → 빌드 → "잘 되는 것 같다"가 현재의 품질 보증 프로세스다.** 이것은 한 명이 관리하는 동안은 작동하지만, 사람이 바뀌거나 6개월 후 코드를 수정할 때 회귀 버그를 잡을 수 없다는 의미다.

## [7] 다음 단계 우선순위

### P0 (3개)

**1. LLM 출력 validation 파이프라인 구축**
- 무엇: 분류기 출력에 토픽 whitelist 검증 + Opus 코멘트에 수치 대조 검증
- 왜 급한가: 깨진 토픽명이 이미 데이터에 존재하고, 수치 오류는 운용보고서에서 치명적이므로
- 달라지는 것: 깨진 토픽 0건 보장 + 수치 변조 감지 가능 → 검토자 부담 대폭 감소

**2. 핵심 경로 regression test 10건**
- 무엇: dedupe, event clustering, salience, fallback, stratified sampling, debate context build의 입출력 고정 테스트
- 왜 급한가: 파라미터 하나 수정이 전체 출력을 변경하는데, 변경 영향을 잡을 수 없으므로
- 달라지는 것: 코드 수정 시 "뭐가 바뀌었는지" 즉시 확인 가능

**3. vectorDB 임베딩 모델을 multilingual로 전환**
- 무엇: `all-MiniLM-L6-v2` → `paraphrase-multilingual-MiniLM-L12-v2` (또는 동급)
- 왜 급한가: 기사 89%가 한국어인데 검색 모델이 영어 전용이라 PA 원인 분석의 뉴스 매칭이 무의미할 수 있으므로
- 달라지는 것: 한국어 기사 검색 품질 정상화 → report 코멘트의 국내 자산 분석 품질 개선

### P1 (4개)

1. **GraphRAG monthly/daily 일관성 수정**: monthly rebuild에 decay/prune 적용, daily에 transmission_paths precompute 추가
2. **수집 소스 다변화**: 국내 경제지 RSS(매경/한경) 추가, NewsAPI 대체 소스 확보
3. **모니터링/알림**: daily_update 실패 시 알림(최소 파일 기반 health check)
4. **evidence trace 개선**: Opus 프롬프트에 article_id 전달 → 출력에 `[ref:ID]` 태그 요청

### P2 (3개)

1. 파라미터 근거 문서화 + sensitivity analysis (salience 가중치, Jaccard 임계치 등)
2. 2025년 데이터 분류 backfill (전체 시계열 분석용)
3. 비용 추적 통합 (분류/GraphRAG/debate 전체 LLM 호출 비용 집계)

→ **아니오.** 파이프라인 구조, 계층 분리, downstream 배선, 비용 관리 설계는 대학생 수준을 넘는다.

**"현업에서 내부 보조도구로 쓸 수 있나?"**
→ **조건부 예.** 조건: (1) 실무자가 수치를 전수 검토한다, (2) LLM 코멘트를 초안으로만 사용하고 그대로 전달하지 않는다, (3) 깨진 토픽명/수치 오류를 수동으로 잡을 준비가 되어 있다.

**"고객용 자동 생성 시스템으로 바로 써도 되나?"**
→ **아니오.** 수치 가드레일 없음, LLM 출력 validation 없음, evidence trace 부재, 모니터링 없음. 무인 운영 시 수치 오류가 고객에게 그대로 전달될 위험이 있다.

---

**현재 위치: 고급 프로토타입**

구조는 내부 파일럿 수준이지만, 품질 보증(validation/test/monitoring)이 프로토타입 수준에 머물러 있어 "고급 프로토타입"으로 판정한다. P0 3건(LLM 출력 validation, regression test, multilingual 임베딩)이 해결되면 "내부 파일럿"으로 올라갈 수 있다.
